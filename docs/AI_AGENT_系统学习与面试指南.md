# AgentKit：企业级 AI Agent 系统学习与面试指南

> 适用对象：有 Python 基础、刚开始做 AI Agent 应用开发，希望能够读懂本项目、独立实现功能并准备技术面试的开发者。

## 1. 学习目标

完成本指南后，你应当能够：

1. 解释本项目从用户请求到工具执行、结果返回的完整链路。
2. 说明 Agent、工作流、RAG、记忆、工具调用、人工审批的边界与关系。
3. 使用 LangGraph 实现可暂停、可恢复、可审计的任务流程。
4. 为 RAG 设计可评测的检索与回答链路，而不是只会调用向量库。
5. 从企业视角讨论可靠性、并发、权限、Token 成本、可观测性和安全。
6. 用本项目中的真实设计与故障案例回答面试问题。

学习时不要只追求“会用某个框架”。企业级 Agent 的核心是：**让模型能力处在可控、可恢复、可评估的系统中。**

## 2. 先理解项目整体

AgentKit 不是单一聊天机器人，而是可部署多个企业业务流程的 Agent 平台。

```text
用户 / Web / CLI
        ↓
TaskRequest（统一任务契约）
        ↓
Runtime（租户、配置、Skill/Pack 装载）
        ↓
Agent Graph（理解任务、路由、执行、恢复）
        ↓
Tool Executor（权限、预算、重试、审计）
        ↓
业务工具 / RAG / 数据库 / 浏览器 / 外部 API
        ↓
结果、审批状态、审计记录、评测数据
```

## 3. 关键概念：Agent 到底是什么

### 3.1 Agent、工作流与聊天机器人的区别

| 概念 | 特征 | 适用场景 |
|---|---|---|
| 普通聊天 | 输入文本，输出文本 | 问答、文案、摘要 |
| 固定工作流 | 步骤预先确定 | 报表生成、表单处理 |
| Agent | 模型根据目标选择工具或路径 | 研究、分析、跨系统任务 |
| 企业 Agent | Agent + 权限 + 审批 + 审计 + 评测 + 恢复 | 真实业务流程 |

一个成熟的 Agent 不应只有“模型调用 + 工具循环”。它至少要回答：

1. 谁可以发起这个任务？
2. 模型可以调用哪些工具？
3. 哪些动作必须人工审批？
4. 工具失败、超时、重复执行后怎么办？
5. 如何知道它做得好不好、花了多少钱？

### 3.2 本项目中的 Agent 设计原则

- **模型负责不确定性，代码负责确定性。** 模型适合意图理解、计划、总结和生成；权限、金额、数据库写入、幂等和重试应由代码控制。
- **副作用必须隔离。** 查询、搜索等读取操作与发布、发送、写库等动作不能使用同样的执行策略。
- **高风险动作必须审批。** 审批不是 UI 功能，而是状态机中的一个中断点。
- **失败不等于可以重试。** 对发布、扣款、发消息等非幂等动作，结果未知时应停止并人工核对。
- **状态要显式保存。** 不要只依赖模型上下文；任务状态、工具结果、审批结果应可恢复。

### 3.3 面试表达模板

问题：为什么不能让 LLM 直接调用所有 API？

回答：LLM 的输出具有概率性，可能产生错误参数、越权操作或重复调用。因此模型只产生候选计划，系统通过工具白名单、参数校验、RBAC、审批、幂等键、审计日志和预算限制把副作用收敛到可控范围。

## 4. LangGraph 学习重点

LangGraph 可以理解为“带状态持久化的 Agent 工作流引擎”。它适合有分支、循环、人工介入、失败恢复的任务，而不仅仅是顺序调用多个函数。

### 4.1 必须掌握的五个对象

| 对象 | 作用 | 在项目中的意义 |
|---|---|---|
| State | 图中节点共享的任务状态 | 保存用户目标、工具结果、审批状态、错误信息 |
| Node | 一个可测试的处理步骤 | 意图识别、检索、生成、审批、发布 |
| Edge | 节点间的固定或条件跳转 | 成功、失败、需要审批、需要澄清 |
| Checkpointer | 保存执行检查点 | 进程重启后继续执行 |
| Interrupt | 暂停图等待外部输入 | 人工审批、补充信息、人工验证 |

### 4.2 用一个发布任务理解状态图

```text
开始
 → 任务理解
 → 读取企业知识 / 搜索
 → 生成草稿
 → 风险检查
 → 等待人工审批
     ├─ 拒绝：结束并返回原因
     └─ 批准：执行发布
                ├─ 成功：记录结果
                ├─ 可重试失败：按策略重试
                └─ 结果未知：停止，人工核对
```

这里的关键不是“画图”，而是每条边都对应一个业务决策。比如“结果未知”不能回到发布节点再次点击，否则可能重复发布。

### 4.3 为什么要用 Checkpoint 和 Interrupt

企业任务常常不是几秒完成：可能等待审批、等待用户补充信息、等待第三方系统，或在执行中发生进程重启。

正确做法是：

```text
节点执行完成
→ 保存状态检查点
→ 遇到审批时中断
→ 审批结果到达
→ 从原状态恢复后续节点
```

这比“把所有上下文放在内存变量里等待”可靠得多。学习时重点观察：中断前哪些状态需要持久化、恢复后怎样避免重复副作用。

### 4.4 LangGraph 练习

1. 画出一个“员工请假审批 Agent”的状态图。
2. 添加 `draft → approval → execute` 三个节点。
3. 在审批节点中断，模拟批准和拒绝两种恢复路径。
4. 给执行节点增加幂等键，证明恢复后不会重复发邮件。
5. 为每条关键路径写单元测试。

## 5. RAG 系统学习重点

RAG（检索增强生成）不是模型训练，而是让模型在回答前从外部知识库检索证据。

### 5.1 RAG 的完整链路

```text
原始文档
 → 解析与清洗
 → 分块 Chunk
 → Embedding 向量化
 → 写入向量库与元数据

用户问题
 → 查询理解 / 改写
 → 向量检索、关键词检索或混合检索
 → 重排 Rerank
 → 组织上下文与引用
 → LLM 基于证据回答
 → 质量评测与反馈
```

### 5.2 先读本项目这些文件

```text
docs/RAG_WORKFLOW.md
core/rag/loaders.py
core/rag/ingest.py
core/rag/store.py
core/rag/retrieval.py
core/rag/service.py
core/rag/eval.py
```

### 5.3 RAG 的常见误区

| 误区 | 正确理解 |
|---|---|
| 有向量库就有 RAG | 向量库只是索引层；切块、元数据、召回、重排和回答约束都决定效果 |
| 召回到文档就一定能回答正确 | 模型可能忽略证据、误解证据或混入自身幻觉 |
| chunk 越大越好 | 太大噪声多、成本高；太小缺失语义，需要根据文档类型评测 |
| 只测最终回答 | 应同时测检索质量、上下文质量、引用质量、延迟和成本 |
| RAG 能解决全部知识问题 | 无相关资料时应该拒答或说明证据不足，而不是编造 |

### 5.4 RAG 与微调的面试答案

| 对比项 | RAG | 微调 |
|---|---|---|
| 主要解决 | 私有、变化快、需要引用的知识 | 风格、格式、稳定行为、特定能力 |
| 更新成本 | 更新文档并重新索引 | 准备数据并重新训练 |
| 可解释性 | 可以展示检索证据 | 通常难以定位知识来源 |
| 企业知识库 | 通常优先选择 | 通常不作为首选 |

### 5.5 RAG 评测指标

至少建立以下指标：

- **Recall@K**：正确文档是否出现在前 K 条召回中。
- **MRR / 排序质量**：正确文档是否排得足够靠前。
- **上下文相关性**：召回内容是否真正与问题有关。
- **答案正确性**：最终答案是否解决问题。
- **忠实性（Faithfulness）**：答案是否能被检索证据支持。
- **引用正确性**：引用是否真实存在且支持该结论。
- **延迟与 Token 成本**：质量提升是否值得额外开销。

## 6. 记忆、上下文与 RAG 的区别

| 类型 | 保存内容 | 生命周期 | 示例 |
|---|---|---|---|
| 当前上下文 | 本轮任务必要信息 | 单次运行或短会话 | 用户当前问题、工具结果 |
| 对话摘要 | 历史对话压缩结果 | 多轮会话 | 用户已经确认的需求 |
| 长期记忆 | 用户偏好或稳定事实 | 跨会话 | 偏好中文、所在行业 |
| RAG 知识库 | 企业文档与外部资料 | 长期、可更新 | 制度、产品手册、FAQ |

学习 `core/memory/` 时要重点关注 Token 预算：不是把历史消息全部塞给模型，而是选择相关记忆、做摘要、设置过期策略，并按租户和用户隔离数据。

## 7. 可靠性：从这次浏览器问题学习

小红书发布场景是企业 Agent 很典型的外部副作用案例。

### 7.1 需要解决的问题

```text
登录态是否可复用？
浏览器为何打开又关闭？
点击发布后如何确认成功？
超时后是否可以重试？
多人或多任务会不会争抢同一个账号？
```

### 7.2 应掌握的设计

- **浏览器 Profile**：保存 Cookie 和登录态，供后续任务复用。
- **浏览器 Session / Lease**：同一次 Agent 运行中复用同一个浏览器 Context，不应每个工具调用都开关浏览器。
- **并发锁**：同一租户、站点、账号档案通常只能由一个任务独占操作。
- **幂等账本**：同一内容、同一发布意图只允许安全执行一次。
- **结果未知状态**：点击后无法确认时，禁止盲目重试，必须人工核对。
- **人工验证边界**：验证码、短信、扫码必须由人处理，系统只能暂停等待。
- **脱敏诊断**：日志记录路径、状态码、截图位置，不记录正文、Cookie、Token 或手机号。

这是你面试时很有价值的真实案例：你可以展示自己理解“Agent 不是 Demo，而是一个与不可靠外部世界交互的系统”。

## 9. 高频面试题与答题方向

### 9.1 为什么企业 Agent 需要人工审批？

当动作不可逆、有资金或品牌风险、涉及敏感数据或对外发送时，模型不能拥有最终执行权。审批应作为持久化状态机的一部分，而不是简单弹窗；审批后恢复同一任务状态，避免重新规划或重复执行。

### 9.2 如何控制 Agent 的 Token 成本？

从四层控制：限制上下文长度；检索只取高质量证据；限制最大工具轮数与最大模型调用次数；为任务、租户和模型设置预算，并记录实际消耗用于优化。

### 9.3 什么时候不应该使用 Agent？

步骤稳定、规则明确、风险很高、要求极低延迟或必须严格确定性的场景，优先用传统工作流或规则引擎。Agent 应只处理确实需要语言理解、规划或开放式决策的部分。

### 9.4 Multi-Agent 是否一定更好？

不是。多个角色拆分会带来额外 Token、协调复杂度、上下文传递和调试难度。只有任务能自然拆分、角色边界清晰、并行收益明显时才使用。很多企业问题先用“一个图 + 多个受控工具/Skill”更稳定。

### 9.5 如何处理工具调用结果未知？

对读取工具可以安全重试；对发帖、扣款、写入外部系统等副作用动作，应记录幂等键和证据，状态标记为 `unknown`，要求人工核对，不自动重复执行。
