# 小红书发布可靠性设计

## 背景与问题

当前小红书图文发布流程已经能够完成登录、图文生成、正文填写和发布按钮定位，但真实平台页面存在以下风险：

1. `xhs-publish-btn` 使用 closed shadow root，真实“发布”按钮不可被常规 CSS 选择器直接定位。当前实现以宿主元素宽度的 61% 作为点击位置，属于经验坐标，不能证明始终命中右侧发布按钮。
2. 标题和正文调用 `fill()` 后没有回读校验。页面重渲染、同页面重复控件或平台状态同步异常时，日志会显示“已填充”，但真正可见表单仍可能为空。
3. 发布后仅以成功文案或 URL 改变确认结果。平台若使用异步接口且保留编辑页，系统会把已点击操作标记为“结果未知”，既无法给出有效诊断，也不能安全重试。
4. `xhs_text_image` 策略创建发布包时不依赖浏览器，但实现仍通过浏览器上下文执行，导致研究完成后出现一次无业务意义的秒开秒关。

本设计只解决发布可靠性和可观测性；不绕过验证码、短信、人机验证或小红书平台限制。

## 目标

1. 在真正点击发布前，保证可见标题和正文与已审批内容一致。
2. 稳定命中 closed shadow root 内右侧“发布”按钮，且记录实际点击几何信息。
3. 在点击后通过页面状态、提交请求/响应和页面截图形成可审计证据；不能确认时保持幂等账本的“结果未知”保护。
4. 对无法确认的直接发布，保留浏览器窗口供人工检查，超时后关闭而不自动重试。
5. 消除图文生成策略创建发布包时无意义的浏览器启动。

## 非目标

1. 不把用户日常 Edge/Chrome 用户目录接入 AgentKit，不复用普通浏览器 Cookie。
2. 不采用第三方 `playwright-skill` 作为生产依赖。该项目可作为一次性开发排障参考，但其任意脚本执行模式不具备本项目所需的审批、租户隔离、幂等和审计边界。
3. 不把单次小红书页面结构探测硬编码为“发布成功”；只有被记录的提交证据和页面成功状态共同满足时才确认发布。
4. 不自动解决验证码、短信或平台审核弹窗。

## 方案比较

### 方案 A：增强现有 Python 连接器（采用）

在现有 `PlaywrightSearchClient` 和 `XhsPublishAdapter` 中补足字段回读、右侧按钮坐标、请求响应观察、失败诊断和人工观察窗口。它复用当前独立持久化登录档案、租户配置、幂等账本和工具审计链路。

优点：没有第二套浏览器运行时；行为可单元测试；租户隔离和审批边界不变。

缺点：需要维护针对小红书的页面适配规则。

### 方案 B：引入第三方 Playwright Skill 作为运行时（不采用）

让模型动态生成 Node.js 自动化脚本来完成发布和确认。

优点：临时排障和页面探索速度快。

缺点：会绕开现有发布账本、审批和工具治理；增加 Node.js 依赖及独立浏览器档案；任意脚本执行不适合作为企业生产发布路径。

### 方案 C：连接用户日常 Edge 浏览器（不采用）

通过远程调试协议控制用户已登录的浏览器。

优点：可直接复用人工浏览器当前页面。

缺点：会混用个人 Cookie、扩展和人工标签页，破坏并发隔离和可追溯性，也难以在服务环境稳定部署。

## 设计

### 1. 表单填写与回读

发布适配器将标题和正文的填写拆为两个步骤：

1. 定位可见的字段并填写已审批文本。
2. 在同一 locator 上回读值，正文兼容 `input_value()` 与 `text_content()`/`inner_text()` 所需的可编辑区域读取方式。
3. 规范化空白后比较预期和实际值。
4. 任一字段不一致时，点击前抛出 `BrowserPageChanged`，并写入页面诊断和截图；不得点击发布。

这将“日志显示已填充”转化为可以证明的发布前置条件。

### 2. closed shadow root 发布按钮

发布宿主仍以 `xhs-publish-btn[is-publish="true"][submit-disabled="false"]` 定位。已确认其 closed shadow root 内包含左侧“暂存离开”和右侧 `button.ce-btn.bg-red` “发布”按钮，单个按钮尺寸为 120×40。

连接器必须：

1. 校验宿主的 `is-publish` 和 `submit-disabled` 属性。
2. 读取宿主边界框并记录 `x`、`y`、`width`、`height`。
3. 以“宿主右边缘向左 60 像素、垂直中心”为优先点击点；当宿主宽度不足以容纳该点时，降级到右侧四分之一区域中心。
4. 点击后记录点击点与宿主边界，供失败诊断使用。

此策略直接利用平台已观察到的按钮尺寸，避免 61% 这种不随结构变化自解释的经验常量。

### 3. 提交结果证据

点击发布前，适配器开始记录与创作中心同源的非 GET 请求及其响应。记录内容仅包含：HTTP 方法、去除 query/fragment 的路径、状态码、资源类型和发生顺序；不得记录 Cookie、请求体、响应体、鉴权头或用户文本。

点击后在配置的浏览器超时内循环检查：

1. 人机验证、手机号验证和登录失效；出现时进入“结果未知”。
2. 页面成功文案、成功 URL、编辑器消失或已发布/审核中的可识别状态。
3. 与提交动作相邻的成功响应。该响应只作为辅助证据，不能单独宣布已发布，直到针对小红书观察到稳定的发布接口路径并加入白名单。

确认成功时返回已发布结果；无法确认时抛出 `XhsPublishOutcomeUnknown`，错误信息附带脱敏网络摘要、页面状态和诊断截图路径。账本继续阻止同一幂等键的盲目重试。

### 4. 人工观察窗口

当直接发布结果未知时，不立刻关闭 headed 浏览器。运行时在配置的有限观察窗口内保持页面，便于操作员查看平台提示或完成人工验证；窗口结束后正常关闭上下文。

默认观察窗口为 90 秒，仅在 `headless=false` 时启用。headless 执行仍立即输出诊断并关闭，避免后台任务悬挂。无论哪种模式，系统都不在结果未知后自动再次点击发布。

### 5. 浏览器生命周期优化

`create_publish_package` 根据媒体策略决定是否需要浏览器：

- `xhs_text_image`：只构造和校验已审批发布包，不启动浏览器。
- `upload` 且已有媒体文件：只构造和校验发布包，不启动浏览器。
- `upload` 且需要生成本地封面：才启动浏览器渲染封面。

研究和真正发布仍使用各自的持久化上下文，保证浏览器进程在每个有副作用的工具调用结束后回收，同时保留独立的站点登录档案。

## 数据流

```text
已审批发布包
  -> 构造发布包（按需启动浏览器）
  -> 打开 AgentKit 独立持久化档案
  -> 生成图文 / 定位编辑页
  -> 填写并回读标题、正文
  -> 记录发布宿主边界，点击右侧发布按钮
  -> 收集页面状态 + 脱敏网络摘要 + 截图
  -> 已确认：写入 published 账本
     未确认：保留观察窗口，写入 unknown 账本，禁止盲目重试
```

## 配置与审计

新增可选配置 `browser_publish_observation_seconds`，默认 90 秒，取值范围 0 到 300。该配置仅控制 headed 运行的结果未知观察窗口，不改变工具总超时、审批要求或幂等策略。

失败诊断文件仍保存在既有发布资源目录下的 `diagnostics/`。日志和异常仅包含经过脱敏的路径、状态和字段长度/哈希，不输出正文、Cookie、手机号、Token 或完整接口响应。

## 测试策略

1. 字段回读不一致时必须在发布按钮点击前失败，并生成诊断。
2. 字段回读一致时继续发布，保持既有发布成功测试。
3. 发布点击必须落在宿主右侧按钮中心区域，不再依赖 61% 常量。
4. 成功页面状态可确认发布；出现验证状态或无确认时均保持“结果未知”。
5. 网络摘要只保留脱敏路径、方法和状态码。
6. `xhs_text_image` 与已有上传媒体的发布包创建不调用浏览器；需要生成封面时仍调用浏览器。
7. 运行既有小红书单元测试、相关集成测试、Ruff 和 Mypy，确认发布账本与审批流程未回归。
