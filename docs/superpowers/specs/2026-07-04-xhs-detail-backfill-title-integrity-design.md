# 小红书详情回填与标题完整性设计

## 1. 背景与问题

小红书搜索页能够返回笔记卡片的标题、链接和可见互动数据，但部分详情链接会跳转到 `error_code=300031` 页面，并显示“当前笔记暂时无法浏览”。当前实现没有把这种“单条笔记不可访问”与登录失效、短信验证或平台风控区分开，导致详情补全提前停止，后续候选也不再尝试。

同时，文案生成和修订处理器会用字符串切片把标题强制截断到租户配置的长度。截断可能破坏语义，并且绕过了现有 Review Gate 对标题超长问题的识别，使修订节点没有机会生成一个完整、合规的新标题。

本设计解决两个相互关联的问题：

1. 在有限的时间和浏览器访问次数内，尽量获得足够的可访问详情，并保留卡片级证据作为降级结果。
2. 保留 LLM 原始标题，由确定性 Review Gate 发现超长问题，再通过一次受控修订生成完整标题，而不是机械截断。

本次修改限于小红书连接器与小红书增长 Skill，不改变通用 Agent、通用 Review Gate 或其他业务 Agent 的语义。

## 2. 方案比较

### 方案 A：显式分类、有限回填、Review 修订（采用）

- 将 `300031` 和“当前笔记暂时无法浏览”等页面识别为单条详情不可访问。
- 单条不可访问时继续尝试候选池中的下一条；只有登录、短信验证和风控挑战才停止整批详情访问。
- 使用搜索阶段已经抓取的候选池进行有限回填，不新增无上限翻页。
- 取消标题硬截断，把长度约束作为生成与修订上下文输入，交给 Review Loop 闭环。

优点是证据质量、稳定性和延迟之间较平衡；错误语义清晰，也便于追踪。代价是详情访问次数会略有增加。

### 方案 B：只修正错误分类和标题截断

仅让单条不可访问不再中止整批处理，并取消标题切片，不扩大详情尝试池。

实现最小、成本最低，但 Top 5 中多条不可访问时仍可能没有足够详情，无法充分利用已经抓取的候选池。

### 方案 C：持续访问直到获得指定数量的详情

不断翻页和打开候选，直到收集到目标数量的可访问详情。

证据完整性最好，但延迟和浏览器调用次数不可控，也更容易触发平台风控，不符合企业级框架的稳定性和成本约束。

## 3. 设计目标

- 一条笔记不可访问不会阻断其他候选的详情补全。
- 登录失效、短信验证和平台挑战仍然立即停止后续详情访问，等待人工处理。
- Top 5 研究使用有限候选池回填，详情尝试次数有明确上限。
- 详情不足时返回卡片级证据和明确告警，不伪造详情内容。
- 标题不再被机械截断；超长标题由 Review Gate 识别，并交给修订节点重写。
- 现有人工发布审批、运行追踪、指标和发布幂等机制保持不变。

## 4. 架构边界

### 4.1 小红书 Playwright 连接器

连接器负责页面状态识别、候选抓取、详情补全和降级元数据，不负责判断文案是否可以发布。

新增专用异常 `XhsDetailUnavailable`，表示当前候选详情不可读取，但浏览器会话本身仍可继续使用。它与以下错误严格区分：

- `BrowserAuthenticationRequired`：登录态失效，停止整批访问。
- `BrowserChallengeRequired`：短信验证、验证码或风控挑战，停止整批访问。
- `XhsDetailUnavailable`：仅当前笔记不可访问，记录后继续下一候选。
- 其他页面或解析错误：记录当前候选失败，继续下一候选，但保留具体错误类型。

页面状态检测同时参考 URL 和可见文本：

- URL 包含 `error_code=300031`；或
- 页面包含“当前笔记暂时无法浏览”“笔记不存在”“笔记已删除”等明确提示。

该状态必须先于通用登录/挑战检测，以免 404 页面中的二维码或导航文字被误判为登录验证。

### 4.2 小红书增长 Skill

Skill 负责研究质量评估、文案生成、Review 与修订。连接器返回的详情可用性属于证据质量输入，不直接等价于 Review 失败。

新增统一的文案约束上下文 `skill.copy_constraints`，至少包含：

- `title_max_chars`
- `body_max_chars`（若租户已配置）
- 标题必须完整表达、禁止从中间截断的规则

生成节点和修订节点都消费同一份约束，避免提示词固定值与租户配置不一致。

### 4.3 通用 Review Gate

通用 Review Gate 不增加小红书特例。它继续执行现有的有限重试模型：

1. 生成候选文案。
2. 确定性规则和 LLM Reviewer 评审。
3. 若可修订，调用小红书 Skill 的 reviser。
4. 达到配置的修订上限后仍不通过，则返回 `blocked`，不进入发布。

## 5. 数据流

### 5.1 搜索与详情回填

1. 搜索页抓取最多 `min(max(limit * 3, limit), 60)` 条卡片候选。
2. 按现有互动指标对完整候选池排序，而不是先截断为最终 `limit`。
3. 详情尝试池上限为 `min(候选数, max(limit, detail_limit) + limit)`。默认 Top 5、`detail_limit=5` 时，最多尝试 10 条。
4. 逐条打开详情：
   - 成功：记录正文、发布时间、收藏和评论等可见详情，并标记 `detail_enriched=true`。
   - 单条不可访问：标记 `detail_enriched=false`、`detail_error=XhsDetailUnavailable`，继续。
   - 登录或验证挑战：标记当前及剩余候选的失败原因，立即停止详情访问。
   - 其他单条错误：记录具体错误类型，继续。
5. 输出时优先选择已补全详情的候选，再用卡片级候选补足数量；每组内部继续按现有互动排序。
6. 最多返回请求的 `limit` 条，并在研究质量中记录详情成功数、失败类型和卡片级降级数量。

这一策略不会为了凑齐详情而继续翻页，因此详情访问次数和整体延迟都有上限。

### 5.2 标题生成、评审与修订

1. 生成节点接收租户文案约束，要求输出完整标题。
2. 处理器只做去除首尾空白，不做长度切片。
3. Review Gate 用确定性规则检查标题长度；超长时产生结构化 finding。
4. 修订节点同时接收原文、Review findings、研究质量和文案约束，重写为完整且不超过上限的标题。
5. 处理器仍不截断修订结果，由第二次 Review 验证。
6. 修订后仍超长或存在其他高风险问题时，流程保持 `blocked`，不会发布。

## 6. 降级与 Review 语义

- `detail_count=0` 表示研究质量为 `limited`，不是连接器级硬失败。
- 仅有卡片证据时，生成内容只能陈述搜索卡片中可见的标题和互动数据，并把延伸内容明确写成原创建议或假设。
- Reviewer 可以因为具体文案包含无证据事实而阻断，但不能仅因为出现 `XhsDetailUnavailable` 就无条件阻断。
- 研究报告必须展示证据覆盖范围和详情不可用告警，便于人工审批者判断。
- 发布前人工审批、发布后核验和幂等账本保持现状。

## 7. 可观测性

现有运行追踪中补充或保留以下字段：

- 候选池大小
- 详情尝试数
- 详情成功数
- 单条不可访问数
- 登录/挑战导致的批次中止原因
- 卡片级降级补位数
- 标题长度 finding 和修订次数

错误值使用稳定的机器可读名称，面向 UI 的说明使用中文。不得把页面正文、Cookie、验证码或敏感登录信息写入日志。

## 8. 测试设计

### 8.1 连接器单元测试

- URL 中 `error_code=300031` 被识别为 `XhsDetailUnavailable`。
- 明确的“笔记不存在/已删除”文本被识别为单条不可访问。
- 单条不可访问只标记当前候选，下一候选仍被打开和补全。
- 登录或挑战仍标记剩余候选并停止后续访问。
- 候选池中前几条不可访问时，可以从后续候选回填可访问详情。
- 详情尝试次数不超过计算出的上限，最终结果不超过请求的 `limit`。

### 8.2 Skill 与 Review 测试

- 生成节点保留完整的超长标题，不再切片。
- 修订节点保留完整输出，不再切片。
- 超长标题由确定性 Review 产生 finding。
- 修订上下文包含租户实际的标题长度约束。
- 一次修订可将超长标题重写为完整合规标题。
- 修订后仍不合规时，Review Loop 按上限返回 `blocked`。
- 详情为零时研究质量是 `limited`，并生成证据范围告警，而不是连接器异常。

### 8.3 回归验证

- 运行小红书连接器、增长 Skill、上下文注册与 Review Loop 的定向测试。
- 运行完整测试套件。
- 运行 Ruff、Mypy 和上下文校验。
- 不执行真实发布作为自动化测试；真实发布仍需显式人工授权。

## 9. 验收标准

- 对包含至少一条 `300031` 的候选集合，系统会继续尝试后续候选。
- 默认 Top 5 请求最多打开 10 个详情，不会无限搜索或翻页。
- 在候选池存在可访问详情时，结果优先返回这些详情，并用卡片结果补足到最多 5 条。
- 日志和结果能区分单条不可访问、登录失效与人工验证。
- 生成和修订处理器中不存在标题长度切片。
- 超长标题经过 Review Loop 重写；无法修复时保持 `blocked`。
- 通用 Review Gate、人工审批、追踪、指标和幂等发布行为无回归。

## 10. 非目标

- 不绕过小红书验证码、短信验证、登录或内容访问限制。
- 不保证每条搜索结果都能读取详情。
- 不新增无限重试、无限翻页或跨账号规避风控机制。
- 不改变其他 Agent 的 Review 策略。
- 不把小红书页面错误硬编码进通用 Agent Runtime。
