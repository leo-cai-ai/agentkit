# 聊天输入框贴底布局设计

## 1. 目标

聊天页采用类似 ChatGPT 的单一工作区：消息区域占据聊天面板的剩余高度并独立滚动，输入框始终位于聊天面板底部。聊天面板下方不再展开业务结果卡片，避免页面高度变化导致输入框离开视口。

## 2. 当前问题

`.ak-chat-surface` 当前声明四行网格，但实际直接子元素只有消息区、运行提示区和输入框三个。输入框落入第三个 `minmax(0, 1fr)` 可伸缩行并被拉高，第四行为空，因此输入框下方出现大块空白。

聊天布局之后还存在 `#result-region`。部分业务响应会将它取消隐藏并在整个聊天面板下方渲染结构化结果，这与输入框始终贴底的单一聊天工作区冲突。

## 3. 布局契约

聊天面板内部固定为三行：

1. `minmax(0, 1fr)`：消息区，独立垂直滚动。
2. `auto`：运行提示或可重试失败按钮；隐藏时不占空间。
3. `auto`：输入框，位于聊天面板底部。

桌面端聊天工作区继续使用当前动态视口高度。移动端使用动态视口单位和安全区内边距，输入框不得遮挡最后一条消息、发送按钮或系统手势区域。

输入框使用布局定位和现有 `position: sticky` 双重约束，但不使用脱离文档流的 `position: fixed`，避免覆盖侧边栏、抽屉或移动端软键盘内容。

## 4. 下方内容处理

- 聊天页上的 `#result-region` 永久保持隐藏，不再在聊天面板下方展示结果卡片。
- 普通业务结果继续以 Assistant 消息展示。
- 计划、审计、原始证据和结构化执行详情继续写入顶部“本轮追踪”抽屉。
- 审批操作继续作为聊天消息内的审批控件展示。
- 其他非聊天页面如果使用 `renderResult()`，保持原有结果区行为。

实现时保留隐藏的 `#result-region` DOM 适配点，避免破坏现有渲染函数与自动化测试；`renderResult()` 在 `body[data-page="chat"]` 上强制抑制主结果面板，但仍更新追踪抽屉内容。

## 5. 滚动与状态

- 新消息、流式 Token 和历史会话加载继续滚动消息区，而不是滚动整个页面。
- 运行提示出现或消失时，只改变消息区可用高度，输入框仍贴底。
- 输入框高度随内容增长到现有最大高度，之后输入框内部滚动，不能把输入框推出聊天面板。
- 本轮追踪作为抽屉覆盖层打开，不在输入框下方插入内容。

## 6. 响应式与可访问性

- 桌面端、平板和移动端均不得产生页面级横向滚动。
- 移动端底部使用 `env(safe-area-inset-bottom)`。
- 输入框、示例问题和发送按钮的键盘焦点顺序不变。
- 消息区保持 `role="log"` 与 `aria-live="polite"`。
- 运行提示保持现有 `role="status"`，不改变屏幕阅读器语义。

## 7. 测试策略

- UI 契约测试验证聊天网格为三行，首行为可伸缩消息区，输入框位于最后一行。
- UI 契约测试验证聊天页强制隐藏 `#result-region`，同时仍更新追踪详情。
- JavaScript 语法检查验证渲染逻辑无语法错误。
- 浏览器在桌面视口和移动视口验证：输入框贴底、消息区可滚动、底部无额外结果内容、页面无横向溢出。
- 运行完整 Pytest 套件防止其他页面结果渲染和历史会话行为回归。

## 8. 验收标准

1. 空会话和长会话中，输入框都位于聊天面板底部。
2. 输入框下方没有空白网格行或业务结果卡片。
3. 消息增多时只滚动消息区，页面主布局保持稳定。
4. 运行提示或重试按钮出现时，输入框仍在底部且不被遮挡。
5. 顶部“本轮追踪”仍能展示计划、审计和原始证据。
6. 桌面端与移动端均无横向溢出，移动端安全区正常。
