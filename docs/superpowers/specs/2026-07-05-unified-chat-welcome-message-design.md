# 聊天欢迎语统一设计

## 背景

聊天页首次由服务端模板渲染时显示中文欢迎语，但点击“新建会话”后，前端 JavaScript 会插入另一条英文欢迎语。切换 Agent 时还存在第三条英文欢迎语。多个硬编码来源导致同一交互在不同入口下语言和语气不一致。

## 目标

- 首次进入聊天页与点击“新建会话”使用同一条 General Agent 中文欢迎语。
- 切换到指定业务 Agent 时使用统一格式的中文欢迎语。
- 欢迎语只有一个配置来源，避免模板与 JavaScript 再次漂移。
- 不改变会话存储、历史消息、Agent 路由和运行时行为。

## 方案

在聊天页根节点或消息线程节点上增加欢迎语数据属性：

- General Agent 欢迎语保存为完整中文文本。
- 业务 Agent 欢迎语保存为包含 Agent 名称占位符的中文模板。

服务端首次渲染继续直接展示 General Agent 欢迎语，前端初始化时读取相同节点上的数据属性。点击“新建会话”或切换 Agent 时，由一个欢迎语读取函数生成消息，再交给现有的 `resetChatThread` 渲染。

General Agent 统一文案为：

> 你好，我负责理解你的需求并协调合适的业务 Agent。你也可以使用 @Agent名称，只指定当前这一轮。

业务 Agent 文案采用：

> 你好，我是{Agent名称}。本轮将由我直接协助你处理相关任务。

若页面数据属性缺失，前端使用同样的中文默认值，保证页面降级时不会重新出现英文文案。

## 测试

- 页面集成测试验证模板包含统一的中文欢迎语及前端读取所需的数据属性。
- JavaScript 契约测试验证“新建会话”和 Agent 切换不再硬编码英文欢迎语，并共同调用欢迎语读取函数。
- 运行相关 Web 测试和静态检查，确认未影响会话历史与消息渲染。

## 非目标

- 不引入欢迎语数据库配置或新增后端 API。
- 不修改其他页面的中英文文案。
- 不改变 `@Agent` 的单轮路由语义。
