# 小红书原生文字图片生成设计

## 根因

现有实现把文章预先拆成 3–8 个 `card_pages`，再尝试在“写文字”页面逐个点击“再写一张”并等待新的编辑器。真实页面行为并非逐页录入：应把完整内容填入第一个输入框，点击“生成图片”，由小红书自动完成封面与正文分页。因此流程在第一张后等待不存在的编辑器增长，最终卡住。

## 设计

发布契约恢复为单一、已审核的 `card_text`：

```python
{
    "card_text": "完整的已审核正文",
    "card_style": "涂鸦",
}
```

`card_text` 默认使用文章正文，也允许 Provider 上游显式传入。内容哈希覆盖 `card_text` 和 `card_style`，确保审批后内容不可被替换。上传图片模式下这两个字段为空。

Playwright 执行顺序固定为：

1. 进入“写文字”；
2. 等待第一个编辑器；
3. 填入并回读校验完整 `card_text`；
4. 等待“生成图片”可用并立即点击；
5. 选择已审核样式并进入最终发布编辑页。

框架不再创建额外编辑器，也不再等待编辑器数量。实际图片数由小红书平台根据内容决定；框架不会承诺精确生成 3–8 张。为提高多页概率，中文内容生成约束调整为适合平台自动分页的完整正文长度，但仍受发布正文最大长度限制。

## 清理范围

- 删除 `xhs_text_image_cards.py` 及其纯分页测试；
- 删除最小页数、最大页数、目标每页字符数配置及租户覆盖；
- Provider、延迟动作、审批预览和前端统一恢复 `card_text`；
- 删除“再写一张”、编辑器计数和逐页校验逻辑；
- 更新已有设计与计划，避免仓库保留相互矛盾的架构说明。

## 错误处理与观测

填充后必须回读验证完整文本；生成按钮不可用、样式不存在或下一步失败时，继续抛出 `BrowserPageChanged` 并保存诊断信息。日志只记录内容字符数、阶段和耗时，不记录全文。

## 测试

1. `card_text` 与样式参与内容哈希，上传模式清空文字图片字段。
2. Playwright Fake Page 只有一个编辑器，断言完整正文被写入后立即点击生成。
3. 测试不得出现添加页面或等待多个编辑器的期望。
4. 审批预览和延迟发布动作携带相同 `card_text`。
5. 上传图片发布链路保持不变。
